主要目的是脱离security本身的表单登录和登录处理逻辑， 毕竟登录过程是可能增加验证码等其他操作的。因为设计到表单的传值改变， 所以不能使用原有的处理+loginSuccessForward的方式。

所以， 就是将框架原有的

  • 前后端不分离的表单提交
  • 使用userDetailService + userDetail对象
  • 增加登录表单属性需要自定义过多的bean， 比如AuthentionProvider， UserDetailService等等

更改为

  • 前后端分离， 前端表单可灵活扩展， 后端api和框架解耦， 参数和处理逻辑灵活扩展
  • 不需要去管UserDetailxxx的模块
  • 登录逻辑的更改不需要考虑过多框架接口

写自定义接口

@RestController
@RefreshScope
public class DemoController {

@RequestMapping("/myLogin")
public String login(HttpSession session,
                    String username,
                    String password
        , HttpServletResponse response) throws IOException {
    //写死用于demo测试
    if (username.equals("user1") && password.equals("123456")) {
        MyUserDetails userDetails = new MyUserDetails();
        userDetails.setUserName(username)
                .setRoles(ImmutableList.of("READ", "WRITE"))
                .setPermissions(ImmutableList.of("菜单1", "表单2", "购买功能"));

        UsernamePasswordAuthenticationToken authentication = new UsernamePasswordAuthenticationToken(username, password, userDetails.toSecurityAuthority());
        authentication.setDetails(userDetails);
        SecurityContextHolder.getContext().setAuthentication(authentication);

        session.setAttribute("userDetail", userDetails);

        return "success";
    }

    return "fail";
}


@RequestMapping("myLogout")
public String logout(HttpServletRequest request) throws ServletException {
    //直接清除掉session和securityContext的内容
    request.logout();
    return "success";
}


@PreAuthorize("hasAuthority('WRITE')")
@RequestMapping("write")
public String write(Principal principal) {
    return "Done";
}


@PreAuthorize("hasPermission(#myUserDetails, '购买功能')")
@RequestMapping("buy")
public String buy(@SessionAttribute("userDetail") MyUserDetails myUserDetails) {
    return "Done";
}

}

索引的原理， 类比于书目录， 从目录-章-节-具体页数如此依次定位。可以理解为范围的切割。

了解索引前， 先回忆一下集中数据结构， 因为mysql有两个索引方法（HASH, B Tree）

HASH

和开发常使用的HashMap一个原理， 将值映射为固定长度的key值， 作为内存的地址，指向到具体对象值。当通过key查找时， 变可通过此地址快速得到其对应的数据。且因为每个地址都是唯一指向一个值， 而内存通过地址检索就是一个快速定位的过程， 所以， where id=1 和 where id=9999 的时间效率是一样的， 即 时间复杂度为 O(1)。

例外情况， hash索引方法在精准查找时效率是不错， 但在范围查询却有不小的短板， 比如 where id > 100 这个条件， hash索引时要怎么找， 只能把全部数据加载出来， 再遍历查找， 基本上没有效率可言， 甚至还可能牺牲了机器性能。

B 树

可分为 B- B B+ 三种， 这里仅聊 B 和 B+ 这两种。

首先是B树

假定B树每个节点都有两个子节点（m==2，即2阶的B树， 也可以叫二叉树）， 左边的节点是比当前节点小， 右边的则是大。如：

每个节点都是具体的数据节点， 当以范围查找时， 比如where id>53, 那么只需要遍历获取 52除左边节点外的其他节点接口， 相比与hash， 范围查找时效率提升可不是一点半点。 且， 当精确查找时， 因为节点左右两边大小顺序固定的问题（类比于二分法查找）， 精确查找的效率也是不差的。

然后是B+树

因为B树每个节点就是一个数据节点， 当查找不能一次命中时， 就会出现多次的磁盘IO， 比如就比如上面的where id=52, 从root节点开始， 读取65， 52<65, 然后读取左边的， 又一次IO。 因为磁盘IO读一个数据和读100个数据消耗的时间是一致的， 故B+树就是B树往IO效率方面改造的一个结构

同样的定义一个2阶的（事实上， 一个3层的树可以表示上百万的数据， 只要定义好）， 和B树不同， 2阶指的是每个节点又2个子节点； 而2阶B+树则是有每个节点又两个子节点， 同时每个节点各自都包含2个元素。

同时， B+树的非叶子节点（最终端的节点）仅持有索引， 叶子节点则是持有所有的数据地址。

因为单节点可以存储大量的索引， 所以B+树可以大幅降低树的层数， 虽说以B树的查询为例， 查询67仍需要读取2个节点（根节点和第三个二级子节点）， 但不妨把数据量放大来看， 仍是3层的数， 但每个非叶子节点都可以存百万个索引， 这是对比于B树， B+树仍然只需要读取2个节点， （两次索引块的IO和一次数据块的IO， 因为叶子节点存在数据， 就算是读取范围树， 也是一次IO，总共就3次~）， 如此看来， 性能提升大不大？

索引建立规避思路

B+树是mysql索引的底层算法之一， mysql提供了两种索引的类型， hash， 和B Tree（即B+树），这时， 针对不同的字段和查询频率已经查询姿势， 就需要针对性地使用不同的索引方法了， 总的来说：

  • hash类型的索引：查询单条快，范围查询慢
  • btree类型的索引：b+树，精准查询和范围查询都有很高的效率，且层数越多，数据量指数级增长（innodb默认支持和使用）

索引分类

  1. 普通索引index :加速查找
  2. 唯一索引（单字段）
    • 主键索引：primary key ：加速查找+约束（不为空且唯一）
    • 唯一索引：unique：加速查找+约束 （唯一）
  3. 联合索引 （多字段）
    • primary key(id,name):联合主键索引
    • unique(id,name):联合唯一索引
    • index(id,name):联合普通索引
  4. 全文索引fulltext :用于搜索很长一篇文章的时候，效果最好。但并不建议在mysql中存超过M基本的文本， 使用es存储甚至更方便， 且模糊查询分词方面极端优秀。

虽说索引不会占据太多的影片空间， 但是当数据量上来时， 索引的空间也是与其成正比的。 所以， 为一个表的各个字段独立建立所以并不是一个很聪明的选择， 尤其时在我们开发时都都不会使用一个字段作为查询条件（id除外）

这里主要对联合索引进行列举使用原则。

最左匹配原则

索引为（name, email, phone）时，

where name=? (√)

where name=? and email=? (√)

where name=? and phone=? (√)

where email=? (×)

不走索引的sql

  1. 模糊查询like 不走索引；
  1. 当or条件中其一字段不在同一联合索引中，不走索引

索引为（name, email）时，

where name=? or email=? or phone=? (×)

  1. not in 不走索引

  2. is not null 不走索引

  3. != <> 不等于不走索引， 类比于第四点， 因为在mysql中， null也是一个值

  4. 类型转换不走索引， 比如 where name=898, 得用括号引起来

  5. 使用order by时，若select 的字段不是索引字段， 不走索引，比如 select gender from table order by email (×)， 而 select name from table order by email (√)， 当然， 对主键排序也是走索引的select * from table order by id

以上只是基本的情况， 具体的sql有没有走索引， 需要各自explain进行观察， 开发时对使用sql进行explain也是一个很好的习惯

顺序id对索引的影响

这就涉及到B+树的插入问题了， B+树有一个特性， 每个节点超过m的时候， 会进行分裂， 当id为顺序递增时， 分裂的节点基本都是在上一个节点的右侧， 所以在索引建立时，基本可以预测到数据移动的时间复杂度等指标。 且叶子节点作为一个链表结构， 在顺序自增的id时，基本上都是在链表最后进行检查插入即可。 当id不是一个顺序自增的值时， 首先， 节点分裂就不可预测了， 万一在最左边的节点上， 其他节点且已经是满员的情况先， 基本会导致所有的节点都得重新分配一遍， 基本上是重建了一整颗树了， 而叶子节点也是同理， 链表的查找和插入也消耗了不少的性能和时间， 就导致那么一个现象： 当数据量到达一个级别时（通常是日志表）会发现insert 一条数据都会奇慢无比。

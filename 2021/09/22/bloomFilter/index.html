Question: 如何从一个大文件（>100G）中高效检查指定字符串所在的行号？

Answer: 要面对两个问题， 大文件读取和字符过滤

基本流程

   graph LR

file[大文件]

reader[文件读取器]

filter[过滤器]

result[结果]

file-->reader-->filter-->|记录所在行号| result

主要要点

  1. 大文件读取

  2. 过滤器的选择

文件读取的方式

因为不可能将100G的内容全部写入内存中， 这个做法是绝不可取的，哪怕服务器真的有100G的空闲内存， jvm的堆大小也不太会设置为100G。 所以常用的Files.readAllLine或者readAllBytes便不可用。那么剩下可以选择的就有：逐行读取，一级NIO。

逐行读取

这是java底层本身对大文件读取的一种支持, 而 readLine 的基本原理是从文件读取 指定容量的内容到内存中， 然后检索是否为一行。 因为bufferReader readline是一种高效的读取， 所以可以直接使用之

nio ByteBuffer

nio 是 替代 java io读取文件的一种方案， 和io面向流不同， nio是面向缓冲区的。和上面可以直接得到一整行的数据不同， 因为没有提供封装好的方法， 所以需要自己对行进行处理

nio MappedByteBuffer (更优解)

这是一个内存映射文件模型， MappedByteBuffer是NIO引入的文件内存映射方案，读写性能极高。这是nio提供的一个读取大文件的方案， 它是直接映射到虚拟内存， 而不是直接内存。另外， 它是一次性读取到虚拟内存， 取回逻辑地址， 和bytebuffer复制到缓冲区不一样。

（与byteBuffer直接读取文件， 性能上飞快的， 这里就不列举了， 本地同样大小的文件跑完两个函数， 用byteBuffer的是8ms， 而MappedByteBuffer 则是微秒级别的）

MappedByteBuffer在处理大文件时的确性能很高，但也存在一些问题，如官方文档中所提到：A mapped byte buffer and the file mapping that it represents remain valid until the buffer itself is garbage-collected.* 打开的文件只有在垃圾回收的才会被关闭（故虚拟内存会在回收前一直占用），而且这个时间点是不确定的。

至此， 读取文件的操作姿势已完成， 之后便是过滤器的选择了。

场景1：输出指定字符串所在的行号

因为上面已经是逐行读取了字符串了， 所以只需要简单地进行字符包含判断，输出行号即可, 稍加改造后

场景2：判断指定行是否重复出现， 比如一个爬虫站点的任务列表

和场景1不同， 此场景需要判断一个元素是否已经包含在某集合中，通常， 我们想到的方法有那么几个：

  • 元素数量不多时， 使用Set集合， 直接判断
  • 元素数量多时， 也是使用Set集合 和set不同， 此集合存储的是元素的hash值， 但存在误判值， 因为两个不同的元素其hash值是可能相同的。
  • 布隆过滤器， 是上面hash值集合的变种， 基本思想就是既然一个hash值存在误判， 那么变使用k个hash函数作为计算标准，降低误判率。

布隆过滤器

其实现原理是： 使用一个超大的位数组和几个(k)hash函数。 位数组是一个二进制的元素数组， 0或1。

  1. 初始化位数组元素为0， 数组长度则根据hash函数的最大值作为指标
  2. 当一个元素被加入到集合时， 首先，s使用k个hash函数将其进行计算， 每个hash函数的值作为位数组的下标， 将改数组下标的值置为1.
  3. 当检查一个元素是否在集合中时， 对此元素进行k个hash函数继续， 得到值，分别去对应的下标检查数组的值， 全部为1，则表示改元素很可能在这个集合； 有任何一个0，则表示此元素一定不会在这个集合。

布隆过滤器在可允许存在小范围误差的场景下效率是很高的。 除了存在误判性外， 还有另一个缺点， 那就是删除问题， 因为删除操作是针对多个下标值进行置0， 但是这个值是可能组成其他元素的下标之一。

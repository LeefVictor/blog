mybatis 作为spring项目中常用的db操作框架， 有必要了解它的两级缓存机制，以及缓存作用的范围和生命周期。

一级缓存

一级缓存是作用在Executor执行器中，每一个SQL 会话都有一个Executor，这个Executor中会持有一个localCache, 当发起查询时，会优先在localCache中进行检索， 命中直接返回，没有命中则进行db查询。这个localcache是仅在这一次会话中生效。

一级缓存是默认开启的， 我们可对一级缓存的操作也只有配置其作用范围：

官方文档对其解析为：
mybatis 使用本地缓存来防止循环引用并加快重复嵌套查询。默认情况下 (SESSION) 在会话期间执行的所有查询都被缓存。如果 localCacheScope=STATEMENT 本地会话将仅用于语句执行，则不会在对同一 SqlSession 的两个不同调用之间共享数据。

缓存key值构成

先看源码

每个queryCache都由mapperId + sql + 查询偏移 + limit + where 参数 + 多环境数据库配置（如果有）组成， 只有这些属性均相同时， 才会从缓存返回数据。
另外， localCached的类PerpetualCache的cache属性也仅仅是一个HashMap。故，map的key也就是上面属性对象的hashcode。

所以，即使是读写分离， 多个读db的前提下， 相同的db结构， 也不会使用到相同的缓存。

缓存清除时机

虽说这个缓存是跟随着SQL session的生命周期的， session关闭了， 缓存自然而然就被回收了。但是， 我们做开发， 经常碰到的情况是同时执行好几个语句， 增删改查都有， 在同一个事务中完成。 那么问题来了， 在同一个会话中

  • ‘select id, name from table where birth > ? limit 10 ‘ 缓存了数据
  • delete from table where id = 19;
  • ‘select id, name from table where birth > ? limit 10 ‘ 是读第一步缓存的数据？

答案： 不是。

update

做法很粗暴， 直接在更新前就干掉了整个Executor的缓存。

commit rollback

同update操作的里， 只要有push到db的操作，不管是成功还是失败， 也是干掉整个Executor的缓存

所以， 基本上， 只要不是查询操作， 缓存的影响基本是不存在的。

query

从方法调用往上找， 可以找到在doQuery时， 也会有清除缓存的操作， 一个是整个会话的sql栈调用完毕， 一个是配置了一级缓存的作用范围。

所以， 最后说一下一级缓存的结论：

  1. 可通过 local-cache-scope配置其作用范围是 SQL会话还是单个语句查询STATMENT
  2. 执行db写数据的操作前， 会先清理掉已有的一级缓存
  3. 一级缓存内部仅仅是一个hashmap， 没有大小，容量等限制
  4. 一级缓存的最大范围是会话级别， 所以多个会话下， 或者 分布式环境下， 读，写高并发时有可能出现脏数据， 可以根据使用环境不同将local-cache-scope设置为STATEMENT

二级缓存

从一级缓存的缓存key组成来看， 即使是读写分离， 多个读db的前提下， 相同的db结构， 也不会使用到相同的缓存， 这一点是一个很大的浪费， 毕竟读写分离的前提下， 数据同步是有主从数据库进行管理， 作为调用方式不需要对这些进行考虑的；且多个操作会话/分布式环境时有可能出现脏数据。

二级缓存原理

从DefaultSqlSessionFactory.openSessionFromDataSource方法往下追踪， 可以看到

只要开启了配置， 就创建一个CachingExecutor, CachingExecutor就是一个装饰器。

  • CacheKey的生成， 仍然以被装饰的Executor为主
  • 具体的DB操作， 仍以被装饰的Executor为主
  • 更新、commit、rollbac时， 对缓存的操作和baseExecutor不相同， 因为作用范围没有限制， 除了因为insert/update/delete等操作进行commit之后，缓存基本不会进行清除或者更新

从上面源码可以看得到， 当commit和rollback操作时， cache也是使用了commit和rollback这两个方法。 二级缓存在commit操作时， 也仅仅是存在一个类似待提交状态的map中， 只有被装饰的executor真正commit时， 才将这些个待提交的对象写入缓存。

同时， 还有一个namespace属性的注解@CacheNamespace， 关于这个配置生效的地方， 可以查看 org.apache.ibatis.builder.MapperBuilderAssistant#useCacheRef 和 org.apache.ibatis.builder.MapperBuilderAssistant#useNewCache

所以， 得到以下结论：

  • 二级缓存仅在写操作和rollback执行时， 清除缓存
  • 二级缓存只有在真正commit前， 才会写入缓存对象中
  • cacheKey 仍然以被装饰的Executor的逻辑为主
  • 仅在本地服务中生效。
  • 可以通过在具体的mapper中指定namespace来确定缓存的范围，使用注解的话是 @CacheNamespace， 相同的namespace下命中的缓存才会生效

故， 开启二级缓存需要考虑以下是否为集群服务和namespace两个因素的影响

二级缓存的缺点已经很明显了，不适用于集群式环境和namespace不同但操作表相同的情况下， 脏数据出现的情况会很大。 对比起使用二级缓存， 直接使用redis等第三方缓存优势反而更大， 缺点就是需要投入精力去实现相应的缓存操作流程。
